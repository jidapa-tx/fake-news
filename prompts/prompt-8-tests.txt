## Running / fixing mode (re-run this prompt any time a test fails)

When code changes cause unit tests to fail, re-run this prompt. In this mode, fix only the tests — do NOT modify application/source code.

**How to run:**
```bash
npm test
npm run test:coverage   # verify ≥80% coverage
```

**Rules:**
1. Source code under `src/`, `app/`, `lib/`, `prisma/`, etc. is the source of truth — do NOT edit it, even if a test failure looks like a bug. If you believe there is a real product bug, list it at the end of your report instead of fixing it.
2. Update test expectations, mocks (`vi.mock()` for Prisma + Gemini), fixtures, and setup to reflect the current behavior of the code.
3. Keep coverage intent intact — if a test asserted behavior X, the replacement must still assert behavior X against the current API shape, not be deleted or weakened to make it pass.
4. Preserve minimum test counts per service (≥5 each) and the >80% coverage target. If coverage drops below 80% after edits, add tests until it recovers.
5. If a service listed above has no corresponding `tests/<name>.test.ts` file, CREATE it with ≥5 tests covering the cases listed.
6. If a test is obsolete because the feature was removed, delete it and note which in the report. Do not leave skipped/todo tests behind.
7. Re-run `npm test` and `npm run test:coverage` until every test passes AND coverage ≥80%. Both green is the exit condition.

**Report at the end:**
- Which test files were changed/created and why (one line each).
- Final coverage numbers (lines/statements/branches/functions %).
- Any suspected real bugs found in source code (file:line + what looks wrong) — flagged, not fixed.
- Final `npm test` summary (passed/failed/skipped counts).
